# Icon Generation as Part of the Vite Pipeline

## Motivation

Bloom's favicon, apple-touch-icon, PWA manifest icons, and brand-asset variants
(`bloom-standard.svg/.png`, `bloom-on-white.png`) are currently generated by two
separate scripts that must be run manually:

- `bin/generate-apple-icon.js` — Node/sharp script rendering `apple-touch-icon.png`
  from `resources/branding/bloom.icon/icon.json` (an Apple Icon Composer file).
- `bin/regen-icons.sh` — bash script that shells out to ImageMagick (`magick`) and
  `python3` to render the favicon/manifest/brand-asset variants from a glyph SVG and
  a background color.

Both sets of outputs are committed to git as static files in `resources/icons/`,
which means they can silently drift from the brand source and require a manual
regen step whenever the glyph or brand color changes.

This work moves generation into the Vite build itself: a Vite plugin generates all
icon outputs fresh on every `dev`/`build` run, from a single declared source of
truth. The outputs become build artifacts (gitignored) rather than committed files.

## Source of Truth

A new config file declares the two brand inputs both generators need:

`resources/branding/icon-config.json`
```json
{
  "glyph": "resources/branding/noun-bloom-5179258-FFFFFF.svg",
  "backgroundColor": "#6A2AAC"
}
```

- `glyph` — path to the transparent-background brand glyph SVG (existing file,
  unchanged; already symlinked into `bloom.icon/Assets/` for the Apple icon source).
- `backgroundColor` — brand color in `#RRGGBB`, replacing the `--bg` CLI argument
  that `regen-icons.sh` previously required.

`resources/branding/bloom.icon/icon.json` remains as the Apple Icon Composer file
(layer/scale/translucency/shadow structure untouched). Its `fill.automatic-gradient`
field is overwritten by the plugin from `backgroundColor` on each run — mirroring
today's `regen-icons.sh --update-bloom-icon` — so the file stays usable if someone
opens it in Xcode's Icon Composer. The apple-touch-icon generator itself no longer
parses this field back out for rendering (see below); it derives its working color
directly from `backgroundColor`, avoiding a lossy round-trip through the P3 string
encoding.

## Generation Modules

Both generators move under `bin/icons/` as plain Node/sharp modules (no external
`magick`/`python3` dependency), each exporting a single async function that takes
the loaded config and writes its outputs to `resources/icons/`.

### `bin/icons/colors.js`

Ports the color-space helpers currently implemented as Python heredocs in
`regen-icons.sh`:

- `hexToDisplayP3(hex)` — sRGB → linear → XYZ → Display P3, formatted as
  `display-p3:r,g,b,1.00000` (for writing into `icon.json`).
- `compensateForAppleRender(hex)` — applies the empirical per-channel darkening
  factors (`[0.602…, 0.4375, 0.796…]`) so the rendered apple-touch-icon matches the
  requested brand color, replacing `compensate_hex_for_apple_render`.

### `bin/icons/generate-web-icons.js`

Replaces `regen-icons.sh`. Renders the glyph composited onto the background color
(and a white background for the "on white" variant) via sharp/SVG, producing:

- `favicon.svg`
- `favicon-96x96.png`
- `favicon.ico`
- `web-app-manifest-192x192.png`
- `web-app-manifest-512x512.png`
- `bloom-standard.svg` (1200×1200, bg-filled glyph — same content as `favicon.svg`)
- `bloom-standard.png` (1200×1200)
- `bloom-on-white.png` (1200×1200, glyph on white background)

### `bin/icons/generate-apple-touch-icon.js`

Adapted from today's `bin/generate-apple-icon.js`. Behavioural change: instead of
reading `icon.json.fill['automatic-gradient']` and parsing the P3 string back to
RGB, it computes its working color as `compensateForAppleRender(config.backgroundColor)`
directly. All existing rendering logic (squircle mask, gradient, glass-glyph filter,
edge glow, corner specular) is preserved unchanged. Before returning, it writes
`hexToDisplayP3(compensatedColor)` into `icon.json.fill['automatic-gradient']` and
persists the file.

- Output: `apple-touch-icon.png`

## Vite Plugin

`bin/icons/vite-plugin.js` exports a Vite plugin that:

- Reads `resources/branding/icon-config.json` once.
- In its `buildStart` hook, runs `generateWebIcons` and `generateAppleTouchIcon`
  in sequence, writing into `resources/icons/`.

This fires for both `vite` (dev server startup) and `vite build`, so generated
files exist on disk before Vite resolves any icon asset request or bundles the
configured inputs.

The plugin is added to the `plugins` array in `vite.config.ts`, alongside the
existing `laravel`, `inertia`, `react`, etc. plugins.

### `vite.config.ts` input list

The explicit icon entries in the `laravel()` plugin's `input` array
(`vite.config.ts:16-21`) are replaced with the full set of generated outputs that
need manifest entries for `Vite::asset()` resolution:

```
"resources/icons/apple-touch-icon.png",
"resources/icons/favicon-96x96.png",
"resources/icons/favicon.ico",
"resources/icons/favicon.svg",
"resources/icons/web-app-manifest-192x192.png",
"resources/icons/web-app-manifest-512x512.png",
"resources/icons/bloom-standard.png",
"resources/icons/bloom-on-white.png",
```

(`bloom-standard.svg` does not need a manifest entry — it's imported directly by
React components via `import bloom from '.../bloom-standard.svg'`, which Vite
resolves as a module asset import rather than through the explicit input list.)

## Git / Cleanup

- Add the generated output filenames to `.gitignore` (scoped to `resources/icons/`)
  and `git rm --cached` the currently-tracked copies. They become pure build
  artifacts, regenerated on every `dev`/`build` run.
- Delete unreferenced cruft found in `resources/icons/`:
  - `apple-touch-icon-original.png` (unused, confirmed via repo-wide grep)
  - `sprouter-standard.svg` (pre-rename leftover, superseded by `bloom-standard.svg`)
  - `spouter-on-white.png:Zone.Identifier`
  - `spouter-standard.png:Zone.Identifier`
  - `sprouter-standard.svg:Zone.Identifier`
  (Windows download-zone artifacts from before the Sprouter→Bloom rename)
- Delete `bin/regen-icons.sh` and `bin/generate-apple-icon.js` (superseded by the
  `bin/icons/` modules).

## Testing / Verification

- Run `npm run dev`; confirm `resources/icons/` is populated and the page `<head>`
  icon links (`resources/views/components/icons.blade.php`) and `AuthorChip`/
  `app-logo-icon` brand imports resolve and render correctly in the browser.
- Run `npm run build`; confirm the same outputs are produced and
  `public/build/manifest.json` contains entries for every icon in the new `input`
  list.
- Hit `/site.webmanifest` and confirm the `icons[].src` URLs resolve to versioned
  build assets.
- Visually compare the generated `apple-touch-icon.png` against the prior output
  to confirm the color/rendering pipeline change (P3 round-trip removal) doesn't
  regress the existing 97.3% pixel-match calibration.
